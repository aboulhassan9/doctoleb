## Design Decisions To Make Before The Next Build Slice

1. Staff invite lifecycle: accepted invite browser login proof is the remaining high-value proof now that create, resend, cancelled-pending reissue, disable/cancel, and reactivation are server-owned.
2. Clinical drafts: tenant DB draft table/RPC with RLS and TTL is now the chosen design. Future draft-like PHI must reuse this pattern instead of browser storage.
3. Tenant automation: v1 remains operator-linked project refs plus the guided runner. Supabase Management API and Vercel REST mutation need a separate provider-executor design with external IDs, cost/region/org controls, and compensation.
4. DB delete policy: keep the zero public DELETE policy invariant unless a future audited service-role RPC has explicit preconditions, postconditions, and compensation semantics.
5. CSP: decide whether to self-host fonts now. Self-hosting makes a strict CSP much easier.

## Current Verdict

The foundation is improving and several concrete defects were fixed in this pass. The biggest remaining risks are no longer in the basic no-domain deploy/boot path; they are in deeper production behavior: staff onboarding completeness, tenant automation, required live DB contract secret configuration, Supabase auth/advisor hardening, rate limiting, backups, browser E2E proof, and production observability.

This system is not ready for a real clinic production launch until the High severity gaps above are closed or explicitly deferred with risk acceptance.
